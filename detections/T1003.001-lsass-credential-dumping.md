# Detection: LSASS Credential Dumping (IN PROGRESS)

**ATT&CK Technique:** T1003.001 - OS Credential Dumping: LSASS Memory
**Tactic:** Credential Access
**Data Source:** Sysmon Event ID 10 (ProcessAccess)
**Severity:** Critical
**Status:** In progress — not yet validated

## Description
Intended to detect processes attempting to access lsass.exe memory, a common technique used to extract credentials from memory (e.g. via Mimikatz or similar tools).

## Splunk Query (draft, untuned)

    index=sysmon EventCode=10 TargetImage="*lsass.exe*"
    | table _time, Computer, SourceUser, SourceImage, TargetImage, GrantedAccess

## Current Status
The dashboard panel for this detection currently returns no results. This is being tracked as a known gap rather than treated as a working detection.

Possible causes being investigated:
- Atomic Red Team test for T1003.001 has not yet been executed successfully in this lab environment
- Sysmon config may not be capturing Event ID 10 (ProcessAccess) for lsass.exe — this requires explicit ProcessAccess rules in the Sysmon config targeting lsass.exe
- Time range on the dashboard panel may not align with when the test was run

## Next Steps
1. Confirm Sysmon config includes a ProcessAccess rule for lsass.exe
2. Re-run Atomic Red Team T1003.001 test and confirm raw events land in the sysmon index
3. Validate and tune the GrantedAccess mask to reduce noise from legitimate processes (e.g. antivirus, EDR agents) that also access LSASS
